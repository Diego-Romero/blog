Engineering & AI

Rewriting My Blog With Claude Code

By Diego Romero · Feb 10, 2026 · 8 min read · engineering ai

There is a particular kind of frustration that comes from owning a tool that is supposed to help you and instead makes everything harder. That was my relationship with the old blog. I'd built it over a year and a half, patching together Tailwind components, wrestling with MDX configurations, fighting with content layers that never quite behaved the way the documentation promised. By the end, opening the codebase felt like opening a junk drawer. I knew something useful was in there somewhere, but I wasn't sure what.

So I decided to start over. And this time, I wasn't going to do it alone.

The Old Blog Was Holding Me Back

The previous blog was a Frankenstein of Tailwind components, MDX configurations, and half-finished content layers. Every time I sat down to write, I spent more time debugging the build system than actually writing. That's a problem — not a small one, either. A writing tool that makes writing harder is not a writing tool. It's an obstacle wearing a writing tool's clothes.

I'd accumulated roughly two hundred pages of drafts that never made it to publish because the friction of getting them from my notes app into the CMS felt like too much overhead. I was more interested in the writing than the publishing infrastructure, and yet the infrastructure kept demanding my attention.

A writing tool that makes writing harder is not a writing tool. It's an obstacle wearing a writing tool's clothes — and the honest thing to do is get rid of it.

— From my notes, October 2025

Choosing Claude Code as My Collaborator

I'd been using Claude for code review and debugging for a while, but this project was different in kind. I wanted to use it as a genuine creative partner — one that understood the design direction, the content model, and the constraints I was working within. Not just an autocomplete engine, but something closer to a thoughtful co-architect.

The experiment was this: I would articulate what I wanted at the level of product decisions and aesthetic goals, and Claude Code would translate that into implementation. I would handle the prose, the taxonomy of ideas, the editorial voice. It would handle the scaffolding, the component structure, the TypeScript interfaces.

"The act of explaining what I wanted, clearly enough for an AI to act on it, turned out to be an act of design in itself."

What I didn't expect was how much the act of explaining my intentions — clearly enough for an AI to act on them — would clarify my own thinking. I had vague feelings about what a good blog should be. Working with Claude Code forced me to make those feelings precise.

The Architecture I Landed On

The final content model is simpler than the previous one by an order of magnitude. Here's the TypeScript that underpins it:

// content-types.ts — the complete content model

interface Post {
  slug:        string
  title:       string
  date:        string          // ISO 8601
  excerpt:     string
  tags:        string[]
  readingTime: number          // minutes
  content:     string          // raw MDX
}

interface Book {
  slug:    string
  title:   string
  author:  string
  rating:  1 | 2 | 3 | 4 | 5
  genre:   string
  pages:   number
  dateRead: string
  review:  string
}

interface Project {
  slug:     string
  name:     string
  platform: string
  desc:     string
  stack:    string[]
  url:      string
  repo:     string
}

Everything else is presentation. The content is just JSON files in a data/ directory. No database, no headless CMS, no content layer. Just typed objects and a build step that reads them at compile time.

What I Learned About Writing and Building

Working on this project changed how I think about the relationship between writing and software. They're more similar than I'd assumed — both are acts of deciding what to leave out. And both get better when you're rigorous about the underlying model before you start worrying about the surface.

  • Clarity of intent is the hardest and most valuable part of any creative project, whether code or prose.
  • AI tools work best as thinking partners, not as vending machines. The quality of the output depends entirely on the quality of the prompt — which is really just the quality of the thinking.
  • Simplicity in architecture pays compound interest. Every decision to not add a feature buys back time for every future session.
  • The constraint of having to explain your aesthetic goals to someone (or something) else forces you to have aesthetic goals in the first place.
  • The blog I want to write is different from the blog I want to build. Keeping those two desires from interfering with each other is an ongoing discipline.

The best tools don't ask you to think about them. They disappear into the background so the work can come forward.

— Something I wrote in my notes the morning I published this post

The new blog is not perfect. No first version of anything is. But it's mine in a way the old one wasn't, and opening it now feels like opening a notebook I actually want to write in.

What This Book Is About

Newport's central argument is deceptively clean: the ability to perform deep, focused work — cognitively demanding tasks carried out in a state of distraction-free concentration — is both increasingly rare in our age of open-plan offices and constant connectivity, and increasingly valuable as the economy rewards those who can learn hard things quickly and produce at an elite level. The collision of those two trends creates an enormous opportunity for anyone willing to take focus seriously.

The book is structured in two parts. The first makes the case for why deep work matters — drawing on the habits of Carl Jung, Mark Twain, Woody Allen, and J.K. Rowling, among others, to illustrate what concentrated effort looks like across different domains. The second part is more practical: four rules for cultivating depth in your own life, ranging from scheduling philosophy to the surprisingly important question of what to do during your breaks.

Newport writes with the confidence of someone who has thought carefully about a problem for years and arrived at conclusions he genuinely believes. There is no hedging here, no performative uncertainty. He thinks distraction is a serious problem and that most people are handling it badly, and he says so clearly. I find that bracing.

Deep work is the superpower of the 21st century. And it can be cultivated — it is not a talent you either have or don't.

— Cal Newport, Deep Work

Why It Resonated With Me

I read Deep Work in January 2025, during a period when I was at my most scattered. I had too many browser tabs open, too many Slack channels to monitor, too many half-finished side projects pulling at my attention. I was busy in the way that makes you feel important but not actually productive — full days with nothing substantial to show for them at the end.

Reading Newport was like having someone articulate a feeling I'd had for years but never put into words: the gap between being busy and doing good work is vast, and most productivity advice talks about the former while pretending to address the latter. I restructured my mornings after reading this book. I started treating the first two hours of my day as protected time — no email, no Slack, no phone. The change in what I was able to produce in those two hours compared to the rest of the day was immediate and significant.

The book also changed how I think about social media, meetings, and "quick" interruptions. Newport doesn't moralize about these things, he quantifies the cost — context switching, attention residue, the slow accumulation of cognitive overhead. The framing helped me make changes I'd tried and failed to make through willpower alone.

What we choose to focus on and what we choose to ignore — plays in defining the quality and meaning of our life.

— Cal Newport, Deep Work

Key Takeaways

  • Schedule deep work sessions like meetings — fixed time, protected calendar blocks, non-negotiable commitments to yourself.
  • Embrace boredom. Reaching for your phone whenever you have a free moment trains your brain to expect stimulation and makes depth harder to achieve.
  • The "any benefit" mindset for social media — using it because there's some benefit — should be replaced with the "craftsman" approach: use a tool only if its benefits substantially outweigh its costs.
  • Shallow work should be batched and time-boxed, not allowed to colonize the whole day in small increments.
  • Evening shutdowns — a ritual that marks the end of the workday — protect rest and reduce anxiety about unfinished tasks.
  • Your ability to concentrate is a skill that degrades without practice and improves with it. Treat it accordingly.

Selected Work

A handful of things I've built and find worth sharing.

diegoromero.blog

Web

A personal publishing platform I've been building, breaking, and rebuilding since late 2024. The current iteration was written in close collaboration with Claude Code — an experiment in AI-assisted design that changed how I think about both software and writing. The goal was simple: a site I actually want to open every day. Editorial warmth, generous typography, no CMS friction.

The architecture is deliberately minimal. Content lives in JSON and MDX files in a data directory. At build time, Next.js reads them, generates static pages, and produces a site that loads instantly and costs almost nothing to host. No database, no content layer middleware, no surprises.

Focus Timer

App

A minimal Pomodoro timer application built after reading Deep Work. I'd tried every timer app on the market — most of them either too precious, too gamified, or too full of features that turned the act of starting a focus session into a five-minute project of its own. This one does exactly what I needed: twenty-five minutes of work, five minutes of rest, a log of sessions at the end of the day.

Session data persists in localStorage so the log survives page refreshes. A weekly review screen shows patterns in when and how long I actually worked — which turned out to be surprisingly useful for identifying the hours of the day when deep work comes more easily.

Reading Tracker

API

A lightweight reading log with a REST API backend that I use to track the books I finish, attach notes and quotes, and assign ratings. The original motivation was practical: I wanted the Reading List section of my blog to be driven by real data rather than hardcoded content. The API became the source of truth that the Next.js build pulls from at compile time.

The database is SQLite, running on a small VPS. For a read-heavy, write-rarely use case like this, it's exactly the right level of infrastructure. The whole system — server, database, deployment — fits in a single repository and takes about two minutes to understand. That's the goal.
