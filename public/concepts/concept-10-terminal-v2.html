Rewriting My Blog
With Claude Code

What happens when you hand an AI agent the keys to your codebase.

tech ai next.js

The Old Stack

My blog has lived in several places. The first was a static HTML file hosted on GitHub Pages — four posts, a contact form that emailed no one, and Comic Sans removed at the last minute. Then came a WordPress install I abandoned after three weeks when I realized I was spending more time managing plugins than writing posts. After that, a Gatsby site that took most of a Saturday to configure and another Saturday to shut down when builds started taking three minutes.

The last version was a Next.js 13 project I was genuinely happy with technically. App Router, TypeScript strict mode, MDX for content, a clean Tailwind setup. It worked perfectly. I published two posts in eighteen months.

The codebase was a museum of good intentions. A dark mode toggle that toggled nothing. A tag filtering system with one tag. A "related posts" feature that always recommended the same post. A reading progress bar that measured the wrong element and filled to 100% on page load. Every time I sat down to write something, I ended up in the codebase instead — extracting a component here, fixing a type there — and never touched the actual content.

The real problem wasn't the stack. I was using code complexity as a form of procrastination. The blog had become the thing I worked on instead of the thing I published to.

Enter Claude Code

I'd been using Claude for code review and rubber-ducking for about a year. The workflow was useful but awkward: describe a problem in chat, get a snippet, paste it in, realize the context was wrong, paste more context back. Each session started from zero.

Then Anthropic shipped Claude Code — an agentic tool that runs in your terminal, reads your files directly, retains memory within a session, and proposes changes you review before they apply. My initial reaction was skepticism. AI agents modifying real files felt like a recipe for subtle disasters that surface two weeks later in production. My second reaction was: I have a codebase I've been procrastinating on for a year. If this breaks it, I wasn't using it anyway.

The first session ran about two hours. I started with a focused prompt: "This is a Next.js blog. I want to redesign the homepage — something that feels like a terminal, dark background, monospace font, amber accent. Don't touch the MDX pipeline or the TypeScript config." What came back wasn't a paste of code to review in isolation. The agent read the existing files, asked one clarifying question, and started proposing changes. I approved most, rejected some with reasons, and by the end of the session the homepage looked like something I'd have spent a full week building alone.

The Workflow

What makes Claude Code different from a chat interface isn't the underlying model. It's persistence and direct file access. Within a session, the agent remembers that you said "never use inline styles" forty prompts ago and applies that constraint automatically. It can run your linter and respond to the output. It can read one component, notice it's inconsistent with a pattern elsewhere in the codebase, and flag the discrepancy without being asked.

The approval flow is what makes this genuinely useful rather than just fast. Every file modification passes through a diff view before committing. I rejected about a third of proposed changes — mostly cases where the agent had the technically correct implementation but the wrong context. These weren't failures; they were the normal cycle of any good collaboration. The agent proposed something reasonable, I had information it didn't have, we adjusted and moved on.

The best way to understand a new tool is to use it on something you care about. I cared about getting this blog out of my local environment and onto the internet.

My internal monologue, February 2026

After several sessions, I noticed the proposals aligning more closely with my preferences. Not because the agent was learning between sessions — each session starts fresh — but because my early feedback had shaped the conventions embedded in the codebase itself. The code had become a specification. Good patterns replicate; the agent read them and followed them.

What I Built

The new blog runs Next.js App Router with TypeScript, Tailwind, and MDX. Terminal-inspired design: dark navy background, amber accents, Space Mono for code-style UI elements, Lora for reading text. After a few sessions with the agent, I landed on this type definition for posts:

TypeScript 
interface BlogPost {
  slug: string;
  title: string;
  date: string;
  excerpt: string;
  tags: string[];
  readingTime: number; // minutes
  published: boolean;
  coverImage?: string;
  bookReview?: {
    author: string;
    rating: 1 | 2 | 3 | 4 | 5;
    isbn?: string;
  };
}

export async function getAllPosts(): Promise<BlogPost[]> {
  const files = await fs.readdir('content/posts');
  return files
    .filter(f => f.endsWith('.mdx'))
    .map(parsePost)
    .filter(p => p.published);
}

The bookReview optional field was the agent's suggestion. I'd mentioned offhand that I wanted to post book reviews alongside regular articles — and without being explicitly asked, it extended the type to support that. A small thing, but it showed the agent reasoning about the larger system rather than just the immediate prompt. That kind of incidental intelligence is what makes the tool feel collaborative rather than mechanical.

The Collaboration Model

I've been asked whether working with an AI agent on code feels like "cheating." It doesn't — and the question reveals a misconception about where programming value actually lives.

The interesting work was never the component architecture. It was figuring out what the site should feel like, what tradeoffs to accept between simplicity and features, what the content should actually be. The agent handled implementation mechanics. I handled judgment. That's a reasonable division of labor — the same division that exists in any effective engineering team.

Software authorship is changing. The question is no longer "can I build this?" — it's "do I understand what I built well enough to own it?"

A thought I had at 11pm on a Tuesday

What surprised me most was how much the approval workflow improved my own clarity. Reviewing diffs instead of writing code forces you to articulate decisions. "I don't like this" becomes "this doesn't match the existing component pattern" or "this introduces a dependency I'm trying to avoid." That precision makes the codebase more coherent — and, incrementally, it makes your own thinking more precise.

What Changed In How I Work

The biggest shift wasn't technical. It was psychological. When the activation energy for shipping drops — when "I want a reading time indicator on posts" becomes a five-minute conversation rather than a two-hour implementation session — the blog stops being something you maintain and becomes something you publish to.

I've shipped more in the past two months than in the previous year combined. Not because the code is better (it's roughly equivalent in quality), but because it stopped being interesting to tinker with. The design works. The performance is fine. The content is the unsolved problem now, and that's exactly where the problem should be. Technical debt doesn't accumulate when you're not using complexity as a delay tactic.

The old version of me would have spent this weekend refactoring the PostCard component. The new version wrote this article instead. Claude Code didn't write it. But it got out of the way so I could.

Ichiro Kishimi & Fumitake Koga

The Socratic Structure

The book arrives as a five-night dialogue between a young man and a philosopher. The young man is angry, certain the world is unjust, and convinced that his past has determined his present. The philosopher listens, nods, and dismantles every argument so gently that you almost miss how radical the conclusions are.

This structure is not a gimmick. Kishimi and Koga chose the Socratic dialogue form deliberately — because the ideas they're presenting resist passive consumption. You need to argue against them. The young man does that for you, and his objections are usually exactly what you're thinking. It reads less like a self-help book and more like sitting in on a philosophy seminar where the professor is infuriatingly calm and always right.

The pacing is slow by design. Each of the five nights introduces a single, disorienting idea, and the nights that follow circle back to test whether you've really accepted it. By night three, I had underlined more passages than in any book I'd read in years.

Adlerian Psychology

Alfred Adler was Freud's contemporary and eventual critic. Where Freud believed the past causes the present — trauma shapes behavior — Adler argued the opposite: we use the past to serve present goals. We don't have a trauma-caused fear of social situations; we cultivate anxiety because it excuses us from risking failure.

This is, to put it mildly, an uncomfortable idea. The book doesn't soften it. The philosopher tells the young man, with genuine warmth, that his unhappiness is a choice — not because he's broken, but because it's currently the safest option available to him.

No matter what has occurred in your life up to this point, it should have no bearing at all on how you live from now on. You are not determined by your past — you determine your life in this moment.

The Philosopher, Night One

I initially resisted this framing because it felt like it was dismissing real suffering. But Adler — as channeled by Kishimi and Koga — is not saying pain isn't real. He's saying the story we build around pain is a construction, and constructions can be rebuilt. That distinction took me several chapters to fully absorb.

Separation of Tasks

The most practically useful concept in the book is what the authors call the "separation of tasks." The core idea: every problem in interpersonal relationships comes from intruding on someone else's tasks, or letting them intrude on yours.

Your task is how you behave. Someone else's task is how they respond to your behavior. You are responsible for the first; you have no business controlling the second. If you're kind to a colleague and they respond coldly, their coldness is their task — not yours to fix, not yours to be hurt by.

All you can do with regard to your own life is choose the best path that you believe in. What judgment other people pass on that choice — that is the task of other people, and is not something you have any control over.

The Philosopher, Night Three

Applied consistently, this is liberating in a way that's hard to describe without making it sound trivial. The book isn't saying "don't care about others." It's saying that care expressed through control is not really care. Letting people own their own tasks is a form of respect.

Who This Book Is For

Honestly? Everyone. But especially anyone who has felt stuck in a story about themselves — the one where your childhood, your circumstances, your personality type explain why things aren't going the way you want. This book doesn't tell you those stories are wrong. It asks whether they're useful.

It's also unusually good for engineers and technical people, who tend to be precise about external systems but surprisingly imprecise about their own psychology. The clarity of the Adlerian framework — all behavior is purposive, all unhappiness is relational, freedom comes through contribution — maps well onto a systems-thinking mindset.

Read it slowly. Argue with it. By the time the young man concedes his final point on night five, you'll probably have conceded a few things yourself.
