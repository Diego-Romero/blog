Rewriting My Blog
With Claude Code

DateFeb 14, 2026 Read8 min TagsENG / AI

The Old Blog Was Holding Me Back

My previous blog was built on a Gatsby template I found in 2019. At the time it felt like the right call — get something live, start writing, iterate later. But later never came. The template's opinions about design accumulated like technical debt, and after three years I was fighting the framework more than I was writing.

The CSS was a graveyard of half-implemented ideas: overridden Tailwind utilities, global styles that conflicted with component-level styles, dark mode implemented three different ways in three different files. The MDX pipeline was held together with deprecated plugins. Every time I opened the repo to write a post, I'd spend an hour firefighting before typing a single word.

I knew I needed to rebuild. I also knew I needed help — not because I couldn't write the code, but because I didn't want the rebuild to become another three-year project.

Choosing Claude Code as My Collaborator

I've been experimenting with AI-assisted development for about a year. Most of that experimentation was autocomplete — Copilot-style tab completion that accelerates the mechanical parts of coding. Useful, but not transformative.

Claude Code felt different from the first session. Instead of predicting the next line, I could describe an architecture and have it reason about trade-offs with me. I could say "I want the content model to work like this, and here's why" and get back a thoughtful response that pushed back where it needed to push back.

"The most surprising thing about working with Claude Code wasn't the speed — it was the quality of the questions it asked back. It caught assumptions I hadn't examined."

Diego Romero, during the rebuild

We spent the first session on architecture alone. Content model, data layer, type safety, rendering strategy. No code written. Just thinking clearly before moving fast.

The Architecture I Landed On

The final stack is Next.js 14 with the App Router, Contentlayer for MDX processing, and Tailwind for styling — but with a custom design system layer that owns all the actual visual decisions. TypeScript throughout.

TypeScript — contentlayer.config.ts 
import { defineDocumentType, makeSource } from 'contentlayer/source-files'

export const Post = defineDocumentType(() => ({
  name: 'Post',
  filePathPattern: `posts/**/*.mdx`,
  contentType: 'mdx',
  fields: {
    title:       { type: 'string',  required: true },
    date:        { type: 'date',    required: true },
    summary:     { type: 'string',  required: true },
    tags:        { type: 'list', of: { type: 'string' }, required: true },
    readingTime: { type: 'number',  required: false },
    draft:       { type: 'boolean', default: false },
  },
  computedFields: {
    slug: {
      type: 'string',
      resolve: (doc) => doc._raw.flattenedPath.replace(/^posts\//, ''),
    },
    url: {
      type: 'string',
      resolve: (doc) => `/blog/${doc._raw.flattenedPath.replace(/^posts\//, '')}`,
    },
  },
}))

export default makeSource({
  contentDirPath: 'content',
  documentTypes: [Post],
})

The content model is strict. Every post needs a title, date, summary, and tags. No orphan posts, no missing metadata. Contentlayer validates this at build time, which means broken content becomes a CI failure, not a runtime surprise.

What I Learned

Three weeks of weekends, maybe forty hours total. The blog is live, the content is migrated, and the codebase is something I'm actually proud of. Here's what stuck with me from the process:

  • AI collaboration works best when you bring the architectural taste and let the AI handle implementation variance.
  • TypeScript strictness isn't overhead — it's the fastest feedback loop available at the content layer.
  • Design constraints are a gift. Starting from a clear typographic system made every decision faster.
  • The App Router's data model is genuinely different from Pages. Give yourself time to think in its terms, not the old terms.
  • Writing about building is different from building. The blog itself is now the lab where I think in public.

"Ship the thing. The second draft of anything is always better than the first draft that never shipped."

Principle I kept returning to during the rebuild

Deep Work
by Cal Newport — 2016

01 / Summary

Cal Newport's central argument is deceptively simple: the ability to focus without distraction on cognitively demanding work is becoming both increasingly rare and increasingly valuable. He calls this capacity "deep work" and contrasts it with "shallow work" — the logistical, administrative, inbox-maintenance kind of work that dominates most modern knowledge workers' days.

Newport divides the book into two parts. The first makes the case for deep work's value through a mix of economic theory, neuroscience, and examples ranging from Carl Jung's tower in Bollingen to modern programmers like John Carmack. The second half is ruthlessly practical: four scheduling philosophies (monastic, bimodal, rhythmic, journalistic), the importance of embracing boredom, quitting social media, and draining the shallows from your schedule.

What distinguishes this from generic productivity advice is the underlying theoretical framework. Newport grounds his prescriptions in deliberate practice research and the idea that attention is a finite cognitive resource that can be trained or squandered.

02 / Impressions

I came to this book already convinced of its thesis, which made the experience both affirming and occasionally frustrating. When Newport is good, he's very good — the chapter on the craftsman mindset and "the any benefit fallacy" of social media is as clear-eyed as any productivity writing I've read. The framework for scheduling depth (time blocks, shutdown rituals, the "Done for the Day" hard stop) is directly applicable and I've since integrated most of it.

"A deep life is a good life, any way you look at it."

Where I found myself pushing back was in the portraits of ultra-deep workers Newport holds up as exemplars. The monastic lifestyle he implies is the ideal works for tenured academics and established craftspeople. For engineers working in collaborative product teams, the rhythmic and journalistic philosophies are more realistic — and Newport does cover these, but they feel like afterthoughts compared to the more romantic, isolation-focused model he seems to personally prefer.

The writing is clear and tight. No chapter overstays its welcome. As a reader I felt respected — Newport assumes you're capable of drawing your own implementation conclusions from well-argued principles.

03 / Key Takeaways

01

Deep work — focused, distraction-free effort on hard problems — produces results out of proportion to the hours spent. It is a skill that can be systematically trained.

02

Time-block your schedule. Don't keep a to-do list — keep a calendar. Assign every hour a job, and when plans change, revise the calendar rather than abandoning it.

03

Embrace boredom deliberately. If you reach for your phone every time there's a pause, you're training your brain to crave distraction, not depth. Boredom is practice for concentration.

04

Apply the craftsman mindset to tool selection: adopt a tool only if its positive impacts substantially outweigh its negatives. The "any benefit" logic for social media is a trap.

05

Shutdown rituals matter. A real end to the workday — including a review of open tasks and an explicit shutdown declaration — allows genuine cognitive rest, which fuels tomorrow's depth.

06

Measure lead indicators, not lag. Track hours of deep work per week. That number is within your control; output and outcomes are not always.
